# Agent Runtime — Native Loop, Sessions, Permissions, Provider Layer

Owns clause families `AGT-*` (loop), `SES-*` (sessions), `PERM-*` (permissions),
`PROV-*` (LLM/provider layer). Phase 6 (walking skeleton) clauses below; Phases 7–10
extend these families per `2026-07-03-standalone-harness-design.md` — new clauses
take the next free number, never reuse.

Terms: a **step** is one assistant turn (one model call plus the tool executions it
requests). A **session event** is one row in the append-only `session_event` table.
The **runtime** is `packages/agent`.

## 1. Provider layer (`PROV-*`)

- **PROV-1.** When the runtime resolves a model reference, it shall resolve through the model registry — the shipped `models.json` overlaid by `~/.obligato/models.json` (user entries win on id collision) — to a configured AI SDK provider instance; an unresolvable reference shall fail with an error naming the reference and listing known model ids.
  *Obligation:* unit — shipped id resolves; overlay id shadows shipped; unknown id throws with both the bad ref and ≥1 known id in the message.
- **PROV-2.** While storing credentials, the runtime shall keep them in `~/.obligato/auth.json` with file mode 0600, written atomically (temp file + rename); when no stored credential exists for a provider, the runtime shall fall back to that provider's conventional env var (`ANTHROPIC_API_KEY`), with stored credentials taking precedence over env.
  *Obligation:* integration — round-trip a credential under a temp HOME, assert mode 0600 and the atomic-write observables (the write goes through temp-file + rename; no temp residue after a save — a crash-injection seam is deliberately out of scope, audit 2026-07-03); precedence test: stored key wins over env var.
- **PROV-3.** When a step completes, the runtime shall compute its cost as integer micro-USD from the registry's per-model prices and the provider-reported usage (all four token classes); when the model has no registry price, cost shall be recorded as unknown — never estimated.
  *Obligation:* unit — fixture usage × fixture prices equals exact expected ints (expected values computed by hand in the test, not by calling `costOf`); unknown-price model yields the unknown marker, not 0.
- **PROV-4.** When `obligato chat` or `obligato run` starts with no configured credential or default model, the runtime shall exit non-zero with instructions to run `obligato auth login` — it shall not probe endpoints or guess a provider. When `obligato auth login <provider>` completes, it shall persist the credential (PROV-2) and the chosen default model to `.obligato/config.json`.
  *Obligation:* CLI integration — unconfigured invocation exits non-zero mentioning `obligato auth login` and makes zero network calls; after a scripted login fixture, the same invocation proceeds past setup.
- **PROV-5.** When resolving anthropic credentials, the runtime shall accept a Claude subscription bearer token (`claude setup-token` output) as a stored credential (`type: "token"`) or from the `CLAUDE_CODE_OAUTH_TOKEN` env var, and shall send it as `Authorization: Bearer` with the OAuth beta header — never as `x-api-key`. Resolution precedence: stored credential (either kind) > `ANTHROPIC_API_KEY` > `CLAUDE_CODE_OAUTH_TOKEN` (Phase 7 decision: setup-token delegation; an own PKCE flow is deliberately deferred — the design doc's ToS-drift risk).
  *Obligation:* unit — the precedence table (all three sources populated → stored wins; env-only combinations resolve in order); adapter fixture with a capturing fetch — a token credential's request carries `authorization: Bearer <token>` plus the OAuth beta header and no `x-api-key`.
- **PROV-6.** While a session runs on a subscription token, step costs shall still be computed from registry list prices (a TPAC yardstick, not spend — EVP §2.1 semantics) and the session shall record `auth_kind: "subscription"` in its session metadata so downstream policy (EVP-7/EVP-9 ledger fence, degradation analysis) can distinguish it; API-key sessions record `auth_kind: "api_key"`, keyless sessions `"none"`.
  *Obligation:* fixture sessions under each credential kind — session_meta carries the matching auth_kind and a priced model still yields non-null list-price costs under a subscription token.
- **PROV-7.** If an anthropic request fails with an authentication error (401) on a subscription token, the runtime shall fail the step with an error naming `claude setup-token` (re-mint path) — no silent retry loop and no fallback to another credential mid-session.
  *Obligation:* mock provider returning 401 under a token credential — the surfaced error names `claude setup-token`; exactly one request was made.
- **PROV-8.** When the runtime assembles a step's model input, it shall emit the system prompt as a `SystemModelMessage` via the SDK's `instructions` option (ai v7 rejects system-role entries inside `messages`; `instructions` is its documented caching path) and mark it and — when the assembled context is non-empty — exactly the final message with an Anthropic prompt-cache breakpoint (`providerOptions.anthropic.cacheControl: { type: "ephemeral" }`, the installed adapter's message-level shape) — the append-only chain (SES-1) makes every step's prefix byte-stable, so all context before the tail breakpoint reads from cache on the next step. The options are provider-namespaced: non-Anthropic providers ignore them and the seam carries no provider branching. Cache-token usage classes and cost flow unchanged (AGT-3/PROV-3).
  *Obligation:* unit — the assembled context for a multi-message chain carries the cacheControl marker on the instructions message and on exactly the final message, and on no other; an empty assembled context (root-only chain) marks only instructions and assembles without error; integration — a capturing-fetch anthropic adapter fixture observes `cache_control` in the outbound request body on the system block and the final message; an openai-compatible fixture's request body contains no `cache_control` and the step still completes.
- **PROV-9.** If a step's model call fails **before its first streamed content part** (a text delta or a tool call — the parts the loop consumes; lifecycle parts never arm the gate) with a transport-retryable error — the adapter's own `isRetryable` classification, or a status in {408, 429, 500, 502, 503, 529}; never an auth/validation 4xx — the runtime shall re-issue the same call up to 2 retries with exponential backoff and jitter (base 500 ms), honoring a numeric `retry-after` header when present (capped at 30 s); the failed attempt appends nothing, so the re-issue is byte-identical (same chain, same assembled context — the append-after-finish ordering makes retry deterministic). A failure after streaming began, an exhausted retry budget, or a non-retryable error fails the step exactly as before (PROV-7's subscription-401 naming unchanged and never retried). Backoff timing and randomness shall be injectable (fixture determinism — the F-126 discipline for stochastic loop behavior).
  *Obligation:* unit — a mock whose `doStream` throws 503 twice then streams: the step completes with exactly one assistant event, 3 provider invocations, and two injected-sleeper calls whose values match the seeded jitter formula; a `retry-after: 7` failure sleeps exactly 7000 ms; a 400 fails with 1 invocation and no sleep; a failure after a streamed delta fails with 1 invocation and no sleep; a subscription 401 still names `claude setup-token` with exactly one request.
- **PROV-10.** When a session's model requests target an endpoint other than the resolved credential's own provider — a base-URL override at session start, a mid-session model switch (`/model`, UX-17) onto an override-endpoint model, or a routing decision (AGT-10) resolving to one — the runtime shall withhold operator credentials from that endpoint on every request: the API key replaced by a dummy that defeats the SDK's env fallback (`obligato-local`, F-119) and no OAuth token sent. Override detection is structural: **any** `base_url` on an anthropic registry entry reads as an override endpoint — the official endpoint is spelled by omitting `base_url`, and an entry that names the official URL explicitly is still withheld (audit pin 2026-07-09: the alternative — allowlisting the official URL — reintroduces a string-comparison surface for the exact leak class this clause closes). Requests to the credential's own provider — including after switching back mid-session — carry the real credential unchanged. (Session-side generalization of EVP-8's eval-run withholding; header discipline per PROV-5.)
  *Obligation:* capture-server integration with real-looking credentials present in the parent env, one case per path — (a) session started against an override base URL, (b) a mid-session model switch onto an override-endpoint model, (c) a routed step resolving to an override-endpoint model: in each, the capture server observed the dummy header **value** (`obligato-local`) and no OAuth bearer header — the assertion is on what the server saw, never on the code branch (F-119 rule); a switch back to the credentialed provider sends the stored credential (second capture fixture asserts its value).
- **PROV-11.** When `obligato auth login openai-compatible` runs, it shall require `--base-url` (the endpoint's OpenAI-compatible root, stored verbatim minus a single trailing slash — no other normalization, never defaulted) and `--model`, and shall validate the endpoint before persisting anything by requesting `GET <base-url>/models` with a 10 s timeout, sending `Authorization: Bearer <key>` when a key resolved and no Authorization header when none did. Key resolution is value-based, at login time only: a non-empty `--key` wins, else a non-empty `OPENAI_API_KEY` env var, else keyless — an empty value reads as absent, the resolved value is what persists, and stored-credential resolution shall never fall back to that env var (the PROV-10 leak class stays closed for third-party endpoints). Probe outcomes partition exactly, status checked before body: an unreachable endpoint (including timeout), an auth rejection (401/403 — even when the body looks like a valid list), a parseable model list lacking the named model (exact, case-sensitive `id` match; `data: []` is a conforming empty list and therefore lacks it), or **any other response status** (fail closed — e.g. 429, 500, 502) shall exit non-zero persisting nothing, reporting on stderr; only 404/405/501, or a 200 whose body does not parse as a model list (a JSON `data` array in which every element bears a string `id`), reads as "endpoint does not implement model listing" and proceeds with a note on stdout. On success it shall persist, in order: one registry overlay entry (`id` = the model, `provider: "openai-compatible"`, the verbatim base URL, `context_window`/`max_output` from `--context`/`--max-output` defaulting 128000/16384, `prices: null` — unknown, never estimated per PROV-3 — `tools: true`), a re-login upserting by id; the key, when one resolved, as a per-entry credential keyed by the model id (PROV-2 file discipline; a keyless login writes no credential and leaves any prior one in place); and the model as the default (PROV-4). (Divergence-pinned 2026-07-11: both blind readers independently committed to fail-closed on unlisted statuses, 401-before-body, empty-list-as-absent, value-based key precedence, single-slash strip, upsert, and a bounded probe; their invented usage-vs-validation exit-code split is deliberately not pinned — `fail()` is uniformly non-zero and nothing consumes the distinction.)
  *Obligation:* CLI integration against a local fixture server — (a) keyed login success: overlay entry written with `prices: null` and the verbatim base URL, credential stored under the model id in an 0600 auth file, config default set, the fixture observed the literal `Authorization: Bearer` key value (assert what the server saw, F-119 rule), and no stream (stdout/stderr) contains the key; (b) named model absent from the fixture's list — and separately a `data: []` response: non-zero exit, no overlay entry, no credential, no config write; (c) fixture answers 401 with a valid-looking list body: same nothing-persisted assertion; (d) fixture with no `/models` route: login succeeds and prints the skipped-check note; (e) keyless login: the probe request the fixture saw carries no Authorization header and the auth file gains no entry for the model; (f) fixture answers 500 — and separately a 206 carrying a valid list — non-zero exit, nothing persisted (2xx-not-200 is fail-closed, not the parse path); (g) env-sourced key: with `OPENAI_API_KEY` set and no `--key`, the fixture saw that Bearer value and it persisted as the credential, and an explicit empty `--key` falls through to the env var the same way; (h) re-login for the same model id upserts: one overlay entry, carrying the second login's fields.

## 2. Agent loop (`AGT-*`)

- **AGT-1.** When the runtime executes a step, it shall make exactly one model call (`streamText`) per step and shall own all loop control — SDK multi-step/continuation features shall not be used; a step that returns tool calls executes them and the next step is a new model call. (PROV-9 amendment 2026-07-05: a transport-failed attempt that appended nothing and is re-issued does not count — "one model call" reads as one *successful* call per step.)
  *Obligation:* unit with a mock provider — a two-step exchange (tool call, then final answer) observes exactly 2 provider invocations; grep-proxy — `packages/agent` sources contain no `stopWhen`/`maxSteps` usage (proxy: SDK loop-control surface, named here; narrow if the SDK renames it).
- **AGT-2.** When a step ends, the runtime shall return exactly one of `continue`, `done`, or `paused(reason)`; while a session is paused, its state shall be fully recoverable from the store — resuming in a fresh process continues from the pause point without re-executing completed work.
  *Obligation:* integration — drive a session to a permission pause, reopen the store in a new object (simulating process restart), resume, assert completion and that pre-pause tool executions ran exactly once.
- **AGT-3.** When a step finishes, the runtime shall ingest one telemetry step event via `safeIngest` carrying first-hand usage (all four token classes), the model id, and cost (PROV-3); if ingestion fails, the session shall continue with a degraded marker (KERN-1 discipline) — telemetry shall never break the loop.
  *Obligation:* integration — a fixture session of N steps yields exactly N step_event rows whose token counts match the mock provider's emitted usage (expected totals computed from the fixture, not re-derived via the ingest path); with ingestion forced to throw, the session still completes and carries the degraded marker.
- **AGT-4.** When the model requests a tool, the runtime shall execute it through the tool registry — exactly the seven core tools in Phase 6: `read`, `write`, `edit`, `bash`, `grep`, `find`, `ls` — with filesystem/process access taken only from the caller-supplied `ToolContext` (`cwd`, `exec`); tool results shall append to the session as events in execution order.
  *Obligation:* integration — a session under a ToolContext rooted in a temp dir writes/reads only inside that dir (bash `pwd` and file ops observed under the temp root); a symlink created inside the workspace that points outside it is refused for `read`/`write`/`edit`/`ls` (containment realpath-resolves each requested path and the `cwd` baseline, so following a link out of `cwd` throws, while a symlinked temp root does not spuriously reject); tool_result events appear in execution order by rowid.
- **AGT-5.** When resume is invoked on a session whose last lifecycle state is not `paused`, the runtime shall refuse with a distinct error naming the actual state and shall append no events. Divergence ruling 2026-07-03: reader B's "the session's last lifecycle state must be `paused`; anything else throws `SessionNotPausedError` and appends nothing" is adopted over reader A's "resume on a done session is an idempotent no-op that reports `done`" — continuing a completed session is SES-4's `--continue` path (a new user message on the chain), never resume.
  *Obligation:* integration — resume on a `done` fixture session raises the distinct error and the `session_event` row count is unchanged (counted before/after); resume on a `paused` fixture proceeds normally.
- **AGT-6.** When a pause is recorded, the runtime shall require a non-empty reason string at record time — divergence ruling 2026-07-03: reader B's "pause reasons are validated at request time (non-empty string)" is adopted over reader A's "an empty-string reason is stored and returned as `paused("")`, not rejected". Paused state shall be derived from the chain, introducing no new event kind — a session is `paused` while its last assistant message requested tool calls (the turn is suspended: an ask unanswered or answered-but-unexecuted, or all results landed awaiting the next model call), `done` when it requested none, and `active` when no assistant message exists yet (audit amendment 2026-07-03: the earlier permission-request-only derivation left step-limit and answered-ask suspensions unresumable, contradicting AGT-2).
  *Obligation:* unit — an empty-string reason is rejected with a validation error and appends nothing; the derivation is discriminating across all three states: unanswered ask → paused, answered-but-unexecuted ask → paused (the AGT-2 resume case), all-results-landed suspension → paused, final answer → done, no assistant yet → active; the kind inventory of a paused fixture introduces no pause-specific kind.

- **AGT-13.** When an `edit` tool call's `old` string has no exact occurrence in the target file, the runtime shall attempt two whole-line-window fallback layers before failing, in order: (a) **trailing-whitespace-insensitive** — a window of file lines equals the `old` lines after `trimEnd` on both sides (replacement lines land as given); then (b) **consistent-indentation remap** — a window matches after full `trim` per line AND the mapping from each distinct `old` non-empty-line leading to its file counterpart is a function (identical `old` leadings always meet identical file leadings — this admits tab↔space substitution and uniform shifts alike, and refuses inconsistent windows); the replacement lines' leadings are rewritten through that same map (a leading absent from the map passes through), preserving the FILE's indentation vocabulary, never the query's. Each layer preserves the occurrence-count discipline: more than one matching window without `all: true` is refused naming the count; `all: true` replaces every non-overlapping window. Mid-line substrings match only exactly (the tolerant layers are whole-line). CRLF normalization is out of scope in v1 — a tolerant replacement lands LF line endings for the replaced window (the untouched remainder keeps its own). When every layer misses, the error shall include a near-miss excerpt — the file lines around the first line whose trimmed text equals the first `old` line's trimmed text, when one exists — so the model can self-correct without re-reading the file.
  *Obligation:* table-driven unit on the edit tool — trailing-WS drift succeeds via (a) with the file's bytes otherwise untouched; a tabs-vs-spaces substitution succeeds via (b) with the file's leading re-applied to the replacement (asserted byte-exact); a window where identical `old` leadings meet different file leadings is refused (inconsistent map); two tolerant windows without `all` are refused naming 2; `all: true` replaces both; a total miss's error carries the near-miss line text; an exact match still replaces byte-identically (layer order: exact first).

- **AGT-14.** When the `grep` or `find` tool executes, it shall prefer ripgrep when present in the session environment — the availability branch rides inside the single shell invocation (`command -v rg`), so each sandbox resolves it for its own filesystem — giving `.gitignore`-aware search (`rg -n --no-heading -H` for grep, `rg --files -g` for find; hidden and ignored files are skipped, which is the intended semantic change from the raw `grep -rn`/`find` fallback that only pruned `.git`/`node_modules`); without ripgrep, the prior fallback behavior is preserved verbatim — including error surfacing: a search error (exit > 1, e.g. an invalid regex) is reported with its exit code and stderr, never as "(no matches)", which is why the cap lives in the tool, not an in-shell pipe (a pipeline would mask the search's exit status). Output shall be capped — 200 lines and a 60,000-character ceiling, applied by the tool — and a capped result shall say so explicitly ("refine the query"), never truncate silently; an exactly-at-cap untruncated result carries no notice. `grep` gains an optional `glob` parameter (rg `-g`, ignored under the fallback with a notice).
  *Obligation:* integration under a ToolContext whose PATH includes rg — a `.gitignore`d file's content does not match while a tracked twin does; the same fixture under a PATH without rg falls back and matches both (fallback preserved, ignore-awareness documented as rg-only; the no-rg PATH is built by excluding the resolved rg directory, never hardcoded); a >200-line result carries the explicit cap notice and an exactly-200-line result does not; an invalid regex surfaces `exit 2` + stderr under both branches, never "(no matches)"; the `glob` parameter narrows matches under rg and produces the fallback notice without it.

- **AGT-15.** When a native session starts, the runtime shall compose its system prompt from exactly three parts: the fixed identity-and-tool-guidance text; an environment block (cwd, platform, ISO date, and — best-effort, absent on failure, never an error — the git branch and dirty-file count when the workspace is a git repo); and the workspace's agent-conventions file (`AGENTS.md`, else `CLAUDE.md`, at the workspace root) when one exists, capped at 8,000 characters with an explicit truncation notice. The composed prompt is recorded once in the session root event (SES-1) — so it is the PROV-8 cache-stable system block and mid-session workspace changes never churn it. `obligato chat`/`obligato run` and the api executor share the one builder (F-085 — no parallel prompt composition).
  *Obligation:* unit — a git workspace with an `AGENTS.md` yields a prompt containing the guidance text, the cwd/platform/date fields, the branch name, and the file's content; `CLAUDE.md` is used only when `AGENTS.md` is absent; an oversized conventions file is truncated at the cap with the notice; a non-git workspace omits the git line and still composes; identity check — the CLI setup and the api executor invoke the same exported builder.

## 2a. Spec-native loop (`AGT-7..9`, Phase 8)

The runtime carries a **SpecContext** loaded from the kernel artifact store: a `governed-file → [clause_id]` map derived from trace links (ART-1) plus the touched clauses' obligation-test paths (`packages/*/test/obligations/<CLAUSE-ID>.test.ts`, the repo convention). When the store has no trace links (an arbitrary user repo), the SpecContext is empty and every spec-native guard below is inert — the loop behaves exactly as Phases 6–7.

- **AGT-7.** After a step whose tool batch modified at least one file the SpecContext maps to a spec clause (PIPE-7 applied to the native loop), the runtime shall, **before evaluating that step's `done`/`continue` outcome**, run each touched clause's obligation test through the caller's `ToolContext.exec` as a **separate** `bun test <that clause's obligation path>` invocation (one exit code per clause, for per-clause attribution), record each executed result as a `session_meta` event carrying an `obligation_check` payload (reusing the kind — pauses/scoped-rules/model-switches set the precedent, AGT-6), and inject any failures into the next step's context. Touched files are identified by the path-declaring tools (`write`/`edit`); a `bash` call in the batch — which can modify a governed file without a declared path — forces a re-check of **every** governed clause, and the content-addressed cache below re-runs only those whose files actually changed (audit F-123: without this a bash-broken governed file would escape the runner and miss the done-gate). A result shall be cached by the pair (clause id, SHA-256 of the concatenation of the clause's governed files in canonical path order) — so a change to **any** of a clause's governed files invalidates its cache, a byte-identical write is a cache hit that re-runs nothing and emits no event, and a prior failure clears by content-addressed lookup (the fix's new hash routes to a new entry; old entries are never mutated or evicted). A touched clause whose obligation-test file does not exist runs `bun test` against the missing path, exits non-zero, and is recorded as **failing** (conservative — a clause that cannot be checked does not pass).
  Divergence-pinned (both blind readers converged, 2026-07-03): "touched clause" for the done-gate is the **session-accumulated** set — every clause touched at any point this session, not only the current step's — and the gate is evaluated on **every** step regardless of whether that step modified a governed file (else the loop could declare `done` by ending on a step that dodges the broken clause). The runtime shall refuse to return `done` while any accumulated touched clause's obligation result at its current governed-file hash is failing; a step that would otherwise be `done` becomes `continue` with the failures surfaced. Obligation-run strictly precedes the done-check — `done` never overrides a same-step fresh failure.
  *Obligation:* integration — a fixture repo with one clause, its governed file, and an obligation test that passes iff the file contains a sentinel: a step that writes the wrong content runs the obligation (one `obligation_check` payload event, failing) and is refused `done` (returns `continue`, failure in the next context); a step that later writes the sentinel passes and reaches `done`; a second unchanged step reuses the cache (no re-run, asserted via an exec spy) and emits no event; a step that modifies a *different* clause's file while an earlier clause is still failing is still refused `done` (accumulated-set gate); a missing obligation file records a fail and blocks; an empty SpecContext never runs obligations and never blocks `done`.
- **AGT-8.** Before executing a `write` or `edit` whose target the SpecContext maps to a clause at tier T1 or above, the runtime shall consult the kernel ART-4 gate (`buildGate`): a stale non-inferred governing clause blocks the tool call (routed as a denied tool result naming spec repair), at T0 it warns and proceeds, an `inferred` clause alerts only, and a write to a **governed area with no non-inferred clause** at T1+ likewise blocks (spec-first — behavior edits need a clause). A recorded human override (`{by, reason}`, persisted via the gate) unblocks any case; headless runs carry no override and therefore block. The block is a tool-result denial (PERM-3 shape), never a crash.
  *Obligation:* integration — a T1 write against a stale confirmed clause is denied with a spec-repair message and the file is unchanged; the same under a recorded override proceeds and persists the override attribution; a T0 stale clause warns and writes; an inferred stale clause writes with an alert; a write with no governing clause at T1+ is denied; an empty SpecContext never blocks.
- **AGT-9.** When a spec-native session ends, the runtime shall emit one `VerificationReport` (the existing schema) into telemetry, populated from the session's obligation results (per touched clause), the repo's conventional test command, open drift events (ART-3), and budget conformance, with `failure_class` set per PIPE-9 when verification failed (`obligation_defect` when only obligations failed, else `code_defect`) and null when it passed.
  *Obligation:* integration — a session that violated a touched clause emits a VerificationReport whose `results.obligations` carries the failing clause with `status: fail` and whose `failure_class` is `obligation_defect`; a clean session emits a report with `failure_class: null`; a session with no SpecContext emits a report with an empty obligations array and null failure class.

## 2b. Live routing & budget (`AGT-10..12`, Phase 9)

The runtime carries an optional **RoutingContext** = a loaded `RoutingPolicy` + `AgentRegistryEntry[]` (kernel `loadPolicy`/`loadRegistry`, from the operator's routing pack). Absent → the session's fixed model and an unbounded budget (Phases 6–8 behavior). All routing/budget/bandit decisions reuse the kernel modules (`route`, `escalate`, `BudgetMonitor`, `exploreDecision`/`recordOutcome`) unchanged — the loop only wires them.

- **AGT-10.** When a RoutingContext is present, the runtime shall route **each step** before its model call: build the feature vector from live session state (`extractFeatures` — `step` from the SDLC stage, `tier` = max tier of the session's touched clauses via SpecContext, `mechanical` from a step-type heuristic, `repo`), call `route()` (recording the decision + vector), and resolve the routed `target` to the step's model via `resolveModel`. On a `mechanical`/T0 step the runtime shall draw bandit exploration (`exploreDecision`, gated by `RoutingContext.explore` — default on) and record the outcome (`recordOutcome`) after the step. Absent RoutingContext, the session's fixed model is used and no decision is recorded. **Tier lag (pinned):** `tier` derives from the session's *recorded* obligation checks, so a governed file's first write — before any obligation check for its clause exists — routes as T0; the tier rises to the clause's on the next step. This is intentional (the loop cannot know a clause's tier before it is touched); the ART-4 write gate (AGT-8), not routing, is what enforces spec-first on that first write.
  *Obligation:* integration — with a two-rule policy (mechanical → cheap target, standard → default) a mechanical step routes to the cheap target and a standard step to the default, each recording a `routing_decision` row with its feature vector; the resolved model id on the step's StepEvent matches the routed target; a mechanical/T0 step records a bandit outcome; no RoutingContext → no routing_decision rows and the fixed model is used.
- **AGT-11.** When a RoutingContext is present, the runtime shall run the session under a `BudgetMonitor` seeded from the routed `budget_tokens`, calling `record(step usage total)` at each step's model-call finish; on the 2× durable pause an **interactive** step returns `paused("budget")` (the operator resolves via `BudgetMonitor.resolve` — `continue` grants one further budget of headroom, `block`/`re_spec` halt), while a **headless** run auto-resolves: up to `BUDGET_HEADLESS_CAP` budget extensions (each a kernel `continue`, the only resolve that clears the pause and grants +1 budget), then `block` (returning `paused("budget:blocked")`). This bounds unattended spend at `(2 + CAP)×` the seed. The pause is durable and resumable (AGT-2 shape) — pause state is derived from the append-only budget-event stream (`isPausedForTriage`), not process state; the in-memory `used` counter resets on a fresh process (an acceptable approximation — the next `continue` re-grants a full budget anyway).
  Divergence-pinned (both blind readers converged, 2026-07-03): the monitor is **seeded once per session** from the routed `budget_tokens` present at session start (the first step's) — later steps' differing budgets never re-seed it; `used` accumulates across all steps, so a session pauses when *cumulative* usage crosses 2× the session seed (not per-step). `record` fires at the model-call finish **before** the step's tool executions, so a step that latches the pause returns `paused("budget")` with its tools not yet run. (Kernel note: the native session budget uses `continue` for its headroom grants because that is the only `BudgetMonitor.resolve` action that clears the pause — the kernel's `escalate`/`block` leave it latched, F-126.)
  *Obligation:* integration — a fixture whose *cumulative* step usage crosses 2× the session-seed budget pauses with `reason: "budget"`, and the budget-event stream carries the 1× overrun, 2× overrun, and triage_requested events in order; a later step's larger routed budget does not raise the pause threshold (per-session seed); a headless run of the same fixture grants exactly `CAP` `continue` extensions then blocks with `reason: "budget:blocked"` (asserting the block fires, not just step_limit); the pause survives reconstruction in a fresh process (resumable — re-derived from the stream without running a step).
- **AGT-12.** When an AGT-7 obligation fails for a touched clause, the runtime shall escalate the next retry step's model via kernel `escalate()` (climbing the routing decision's escalation ladder, recording the escalated decision) rather than silently retrying on the same model; when the ladder is exhausted the escalation resolves to triage. Absent a RoutingContext, obligation failures still block `done` (AGT-7) but no escalation is recorded.
  *Obligation:* integration — with a routing policy whose escalation ladder is [cheap, strong], a step that writes content failing its obligation escalates the retry to the next ladder target (a new `routing_decision` with `kind: "escalation"`, `attempt: 1`), and the retry runs on that target; ladder exhaustion records a triage escalation; no RoutingContext → the obligation still blocks `done` with no routing_decision recorded.

## 3. Sessions (`SES-*`)

- **SES-1.** While persisting session history, the runtime shall append rows to the `session_event` table (ULID id, nullable `parent_id`, kind, JSON payload, UTC timestamp) and shall never UPDATE or DELETE them; reads shall order by `rowid` (F-060/F-067 convention).
  *Obligation:* unit + grep-proxy — the session store module issues no UPDATE/DELETE against `session_event` (source scan, proxy named); a written event is immutable via the store's public API.
- **SES-2.** When the runtime reconstructs context for a step, it shall walk the `parent_id` chain from the session head to the root and reverse it; reconstruction shall be deterministic — identical chains yield identical contexts.
  *Obligation:* PBT — for any generated event chain (fast-check), reconstruction returns exactly the chain's events root-first; two walks of the same chain are deeply equal.
- **SES-3.** When an event is appended, the runtime shall record the new head as a `head_moved` event; the current head shall be derived as the most recent `head_moved` by `rowid` — no mutable head column.
  *Obligation:* unit — after a sequence of appends the derived head is the last appended event; the migration adds no head column to any table (schema introspection).
- **SES-4.** When `obligato chat --continue <session>` or `obligato run --continue <session>` starts, the runtime shall load the session's head and continue the same event chain; when a new session starts, the runtime shall create a kernel session row (`startSession`) so TEL-5 markers and lockfile pinning apply to native sessions.
  *Obligation:* integration — `obligato run --continue` (the operator surface, F-085) extends the same session: the follow-up user_message's parent is the prior head and no second kernel session row appears; a new session produces a kernel `session` row with runner metadata; a session that reached `done` on a text-less, tool-less final assistant turn is still `--continue`-able — reconstruction drops empty-content assistant messages so the provider accepts the resumed chain rather than rejecting an empty message.
- **SES-5.** When a session row is created, the creator shall record its runner in the kernel `session.runner` column — `"native"` for agent-runtime sessions, `"cc"` for Claude Code hook sessions; rows created before the column existed read back null. Downstream telemetry math stays runner-blind (AGT-3); only degradation analysis and EVP publishing policy read this column.
  *Obligation:* migration adds the nullable column additively (pre-existing fixture rows read back null); a native session's row carries `runner = 'native'`; the cc-plugin session-start hook fixture writes `runner = 'cc'`.

## 3a. Tree sessions & session→eval (`SES-6..8`, `EVP-10`, Phase 10)

- **SES-6.** When `obligato session fork <session> [event-id]` runs (default `event-id` = the session's current head), the runtime shall append a new `head_moved` event whose payload head is a fresh **fork marker** event parented at `event-id`, producing a branch that shares every ancestor of `event-id` while leaving the original head reachable (SES-3: head is the latest `head_moved` by `rowid`, so both branches' heads coexist). Reconstruction from either head is deterministic (SES-2): the forked branch's context is exactly the ancestors of `event-id` plus new events appended after the fork; events on the *other* branch after the fork point are absent. A snapshot recorded at session start (`storeSnapshot`, best-effort — null when the repo is not a git working tree) is inherited by every fork of that session.
  Divergence-pinned (both blind readers converged, 2026-07-03): fork appends the marker (parent = E) then a `head_moved` to it, so the fork **becomes the current head** (latest `head_moved` by rowid); the original head is *preserved but no longer current* — recoverable by the id `forkSession` returns (or the pre-fork `head_moved` row), append-only so never lost. E is included in the fork's context (the shared seam); the fork marker is the context tip (inert `session_meta`); every subsequent append attaches to the marker, keeping the original branch's reconstruction untouched. Snapshot inheritance is structural (the root snapshot is every branch's null-parent ancestor).
  *Obligation:* integration — fork at an interior event E: the new head reconstructs to E's ancestors only (events appended to the original branch after E are absent from the fork's context), the original head still reconstructs to its full chain, and both heads are simultaneously derivable; a fork of a session with a recorded snapshot carries the same snapshot hash; fork at a non-existent event id errors and appends nothing.
- **SES-7.** When `obligato session compare <headA> <headB>` runs, the runtime shall reconstruct both chains, compute each branch's accumulated cost (sum of `cost_micro_usd` over its assistant messages) and outcome (its lifecycle plus last assistant text), identify the divergence point (the deepest common ancestor by `rowid`), and report the per-branch cost and outcome with the shared-prefix event count. Comparison is read-only — it appends no events.
  *Obligation:* integration — two branches forked from a common point: compare reports the common-ancestor event id, each branch's summed cost (a cheaper branch is identified as such), and each branch's last text; identical heads report a zero delta and the full chain as the shared prefix; the store is unchanged after compare (no appended events).
- **SES-8.** When `obligato session compact <session>` runs, the runtime shall summarize the reconstructed chain up to the head on a cheap routed model (or a caller-supplied summarizer), append a `compaction` event carrying the summary text and the `[from_event, to_event]` range it covers, and thereafter reconstruct context by substituting the summary for the covered span. A fork parented at an event **before** a compaction retains the full pre-compaction history (the compaction is an event on one branch only, SES-6). Compaction appends exactly one event and never deletes the covered events (SES-1 append-only).
  *Obligation:* integration — after compact, reconstruction replaces the covered span with one summary event and the covered originals remain in the store (listEvents still returns them); a fork before the compaction reconstructs the full original history (no summary); compact records the covered `[from,to]` range and one `compaction` event.
- **SES-9.** When `storeSnapshot` returns — success or error — no temporary bundle file created by that call shall remain outside the snapshot store; a cleanup failure shall not fail an otherwise successful snapshot.
  *Obligation:* unit — snapshot a fixture repo with the OS temp location observed: after success the temp location has no new entries and the stored snapshot still restores; with a fault-injected copy-into-store failure the error propagates and the temp location still has no new entries.
- **EVP-10.** When `obligato promote <session>` runs, the runtime shall compile the session into a staging `BenchmarkTask`: `statement` = the first user message, `snapshot` = the session's recorded snapshot hash (erroring if absent — a promotable session must have captured one), `checks` = one `obligations` check plus one `command` check per distinct governed clause the session touched (from its `obligation_check` events), `budget_ceiling_musd` = the session's actual accumulated cost × 1.5, and write it into the staging suite. Replay is the existing `runEval` over that task with `executor: "api"` (Phase 7); promotion itself runs no model. Promotion of a session with no snapshot, or with no first user message, errors and writes nothing.
  *Obligation:* integration — a finished session with a snapshot and touched clauses promotes to a BenchmarkTask whose statement is the first user message, whose budget ceiling is the session cost × 1.5, whose checks include the touched clauses, landing in the staging suite; a session with no snapshot errors and writes no task; the promoted task validates against the BenchmarkTask schema and is runnable by `runEval` (executor api) to a verdict.

## 4. Permissions (`PERM-*`)

- **PERM-1.** When a tool call is requested, the runtime shall evaluate rules `{tool glob, arg glob?, action: allow|ask|deny}`: any matching `deny` rule shall win regardless of specificity (divergence ruling 2026-07-03 — both blind readers noted that under specificity-first, "a blanket `{tool:"*", arg:"/etc/*", deny}` guard is defeated by any tool-literal `allow` rule"; deny is a guardrail, so it trumps). Among the remaining matching rules the most specific shall win, scored by the lexicographic tuple `(literalChars(toolGlob), literalChars(argGlob))` — literal characters are those that are not `*` or `?`, an absent arg glob contributes 0, and the tool component strictly dominates; exact-tuple ties resolve `ask > allow`; rule-list order shall never decide between different actions. Globs are flat: `*` matches any run of characters **including `/`**. When no rule matches, the default shall be `allow` for read-only tools (`read`, `grep`, `find`, `ls`) and `ask` for every other tool — `write`, `edit`, `bash`, and any tool outside the core set.
  *Obligation:* table-driven unit — deny-trump case (`{tool:"*", arg:"/etc/*", deny}` beats `{tool:"read", allow}` on `read("/etc/passwd")`); specificity ordering among allow/ask rules incl. tool-dominance (`(4,0)` beats `(0,5)`) and arg discrimination at equal tool specificity; the `ask > allow` exact-tie rule; flat-glob `/`-crossing; each default including an unknown tool; expected outcomes enumerated by hand in the table.
- **PERM-2.** When a rule resolves to `ask`, the runtime shall append a `permission_request` event and pause the step (AGT-2); the answer shall append a `permission_decision` event, and an "always allow" answer shall additionally append a session-scoped allow rule as an event — never a config-file write. The scoped rule is tool-granular (it matches the surface's "always allow <tool>" wording; deny rules still trump it, PERM-1).
  *Obligation:* integration — an ask/answer/always flow leaves request, decision, and scoped-rule events in order; a subsequent identical tool call proceeds without a new request; config files unchanged (hash before/after).
- **PERM-3.** While running headless (`obligato run`), an `ask` resolution shall resolve to `deny` unless an explicit allow flag was passed; a denied tool call shall return an error result to the model (the step continues — denial is feedback, not a crash).
  *Obligation:* CLI integration — `obligato run -p` with a write-requesting fixture: without the flag the write is denied, the file does not exist, and the session still reaches a final message; with the flag the write occurs.
- **PERM-4.** When a rule resolves to `ask` on an interactive surface, the rendered prompt shall name the tool, its primary argument, and the ask's provenance — the matched rule (tool glob, arg glob, action) or the literal token `default` when no rule matched (PERM-1 defaults) — and the appended `permission_request` event's payload shall carry the same provenance, so a recorded decision is auditable without re-running rule resolution. Credential values shall never render in the prompt (UX-19 discipline).
  *Obligation:* integration — an ask flow under a fixture rule set: the rendered prompt contains the tool name, primary argument, and the matched rule's globs; the stored `permission_request` payload carries identical provenance; an unmatched-default ask renders and records the literal `default`. (`SessionEvent.payload` is an open record by design — the provenance field extends no per-kind schema, so no roundtrip arbitrary changes.)
